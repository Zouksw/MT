## C1:进程内测试重构

### 根因(经 3 个 Explore agent 调研确认)
集成测试用 `request("http://localhost:8000")` 打**生产后端**,与 Redis 共享状态 → 限流/authLockout 触发 **429 假失败**。旧模式下"通过测试"实际是 `dbAvailable=false` 自行跳过。

### 实施内容
| 文件 | 操作 |
|------|------|
| `src/app.ts`(新建,276 行) | 从 server.ts 抽取纯 `createApp()` 工厂,返回 `{app, httpServer, io}`,**无副作用** |
| `src/server.ts`(重写) | 变薄为启动器:调 `createApp()` → `start()`(listen + scraper + cron)。所有副作用移入函数,`import` 不触发后台任务 |
| `src/test/helpers/testApp.ts`(新建) | 共享 `createTestApp()`/`getPrisma()`/`isDbAvailable()`/`getAdminToken(app)`/`SEED_ADMIN`,消除 9 个文件的复制粘贴 |
| `src/test-setup.ts`(改) | `DATABASE_URL` 从空库 `mt_test` 改为种子库 `mt_db`(进程内 app 的 prisma 读此变量) |
| 10 个测试文件(改) | `request(BASE)` → `request(app)`;删除重复的 `checkDatabase()`/`REAL_DB_URL`/`BASE`/`PrismaClient` import |

### 关键技术决策
1. **`createApp()` 返回三元组** `{app, httpServer, io}`:测试只需 `app`,启动器需 `httpServer.listen()`。
2. **4 个顶层 `setInterval` 必须移入函数**:否则 `import { createApp }` 即触发后台任务(订单检查/数据刷新),破坏测试隔离。
3. **`DATABASE_URL` 必须指向 `mt_db`**:进程内 app 的 `@/lib` prisma 单例读此变量;原 `mt_test` 是空库,导致 admin login 失败(根因之一)。
4. **共享 prisma 单例**:多个 suite 复用一个 PrismaClient,`afterAll` 里**不能** `$disconnect()`(会断开影响其它 suite)。

### 验证结果
```
集成测试: 143 passed | 1 failed | 0 个 429  (旧模式: 大量 429 假失败)
单元测试: 68/68 passed
后端 tsc: 0 错误 | build: 成功
```
**唯一失败** "should list existing datasets":`datasets` 表 0 行(无种子数据),测试断言 `toBeGreaterThan(0)`。预先存在的数据依赖,非 C1 引入。

### 消除的重复
9 个文件逐字复制的 `checkDatabase()`(fetch /health + PrismaClient)→ 统一为 `testApp.ts` 的 `isDbAvailable()`(进程内无需探活 HTTP)。
